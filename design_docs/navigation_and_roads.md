# Дизайн-документ: навигация и дороги

## Назначение

Система определяет, по каким клеткам могут передвигаться жители и транспорт, как
выбирается маршрут и как покрытие меняет его стоимость и фактическую скорость.
Дороги являются данными покрытия, а не вторым маршрутизатором: маршрут всегда
строится только через `NavGrid` и `GridRouteService`.

Цели системы:

- жители не проходят через здания, воду и опасные участки;
- более дешёвое покрытие естественно притягивает маршруты;
- изменение препятствий или покрытия немедленно делает старые маршруты устаревшими;
- работа остаётся предсказуемой при большом числе жителей.

## Текущее состояние без дорог

Уже реализованы следующие компоненты:

- `NavGrid` хранит размеры клеток, блокировки и разреженные переопределения веса;
  клетка без переопределения имеет вес травы `2.0`.
- `GridRouteService` использует взвешенный A* по восьми направлениям. Диагональ
  имеет длину `sqrt(2)` и запрещена, если хотя бы одна из прилегающих
  ортогональных клеток заблокирована.
- A* использует бинарную очередь приоритетов и допустимую octile-эвристику,
  умноженную на минимальный вес сетки.
- После поиска weighted string pulling удаляет изгибы, только если прямой отрезок
  не пересекает препятствия и стоит не более чем на 8% дороже заменяемого участка.
- `NavGrid.segment_cost()` использует Amanatides & Woo и проверяет обе клетки при
  пересечении угла, поэтому путь не срезает угол здания.
- Скорость пешехода равна `base_speed / cell_weight`; трава с весом `2.0` даёт
  50% базовой скорости.

`NavGrid` владеет монотонным `revision`. Он увеличивается только при реальном
изменении размеров, блокировок или весов. Каждый `RouteResult` содержит revision,
при котором он был рассчитан. Актор гражданина перепланирует путь до следующего
шага при несовпадении revision.

Недостижимый маршрут повторяется с экспоненциальной задержкой от 2 до 16 секунд.
Изменение revision отменяет ожидание и разрешает немедленную попытку.

## Модель стоимости

Для перехода из клетки `A` в соседнюю клетку `B`:

```text
cost(A -> B) = geometric_distance(A, B) * traversal_weight(B, profile)
```

Вес всегда положителен. `1.0` является базовой скоростью пешего движения;
значения меньше `1.0` допустимы для улучшенных дорог. Блокировка не выражается
весом: такая клетка вообще исключается из поиска.

Базовые значения для будущего покрытия:

| Покрытие | Вес | Скорость пешехода |
| --- | ---: | ---: |
| Трава | 2.0 | 50% |
| Песок, кустарник | 3.0–4.0 | 25–33% |
| Болото, густой лес | 8.0 или запрет | 12.5% |
| Тропинка, грунтовая дорога | 1.0 | 100% |
| Глинобитная | 0.9 | 111% |
| Деревянная | 0.85 | 118% |
| Каменная | 0.8 | 125% |
| Асфальтовая | 0.7 | 143% |
| Асфальто-бетонная | 0.6 | 167% |

Игровые коэффициенты из таблицы являются начальными значениями баланса. Скорость
и стоимость должны происходить из одной таблицы покрытия, чтобы маршрут не
предпочитал поверхность, на которой затем движение неожиданно медленнее.

## Контракт маршрутизации

Новый публичный API должен развиваться от текущего `find_route(from, destination)`
к типизированному запросу:

```gdscript
class_name RouteRequest
extends RefCounted

var from: Vector3
var destination: Vector3
var traveler_profile: StringName = &"pedestrian"
var allow_destination_cell: bool = false
```

`traveler_profile` обязателен до появления велосипедов, тележек и машин. Он
задаёт доступность покрытия и его вес. Например, тропинка допустима для пешехода,
но не для тележки; грунтовая дорога допустима для тележки, но может иметь штраф
для машины в дождь. Нельзя кодировать эти ограничения только в визуальном типе
дороги или в логике актора.

`RouteResult` содержит `reachable`, `waypoints`, `arrival_position` и
`grid_revision`. Пустой список waypoints никогда не означает успешный маршрут.

## Владение данными

`NavGrid` — единственный источник проходимости и итогового веса, который читает
`GridRouteService`. Будущая `RoadNetworkService` владеет семантикой дорог,
строительства и трафика, но применяет изменения через API `NavGrid`:

```text
RoadNetworkService -> NavGrid.set_cell_weights(overrides) -> GridRouteService
```

`NavGrid` хранит только отличия от `DEFAULT_CELL_WEIGHT`. Bootstrap обновляет
только блокировки; он не должен заново заполнять все клетки значением травы,
иначе уничтожаются дорожные overrides и растут затраты поиска.

Routing следует вынести из `world/application` в отдельную feature `routing`,
когда появятся дороги и профили транспорта. Это соответствует
`docs/architecture.md`: выбор маршрута не должен зависеть от `Citizen`, UI или
bootstrap-контроллера.

## Дороги

| Тип | Пешеход | Транспорт | Эра |
| --- | ---: | ---: | --- |
| Тропинка | 100% | запрещён | `Era.TENT` |
| Грунтовая | 100% | велосипеды, тележки | `Era.EARTH` |
| Глинобитная | 110% | грузовые велосипеды | `Era.CLAY` |
| Деревянная | 120% | мопеды | `Era.WOOD` |
| Каменная | 130% | ранний лёгкий транспорт | `Era.STONE` |
| Асфальтовая | 140% | лёгкий моторный транспорт | `Era.BRICK` |
| Асфальто-бетонная | 150% | тяжёлый транспорт | `Era.INDUSTRIAL` |

При ручной постройке клетка остаётся исходным покрытием до завершения работ.
После завершения `RoadNetworkService` меняет override веса, визуал и revision
сетки одной транзакцией. Улучшение дороги использует тот же процесс и не меняет
проходимость мгновенно.

## Desire lines и трафик

Трафик начисляется только при переходе жителя в новую клетку, не на каждом кадре
и не на каждом физическом шаге. Это исключает зависимость от FPS и скорости
симуляции.

```text
пересечение границы клетки
  -> register_cell_entry(cell, vehicle_profile)
  -> traffic[cell] += profile_traffic_weight
  -> при пересечении порога: запланировать изменение покрытия
```

Тепловая карта разреженная. Раз в игровые сутки значения умножаются на
`1 - decay_rate`; значения ниже epsilon удаляются. Для автопревращения в тропинку
и деградации нужны разные пороги и задержка в несколько дней, иначе покрытие будет
мигать около границы:

- `path_create_threshold` создаёт тропинку;
- `path_degrade_threshold < path_create_threshold` начинает деградацию;
- деградация завершается только после `N` подряд игровых дней ниже порога.

Автоулучшение грунтовой дороги создаёт обычный строительный заказ с ограничением
дневного бюджета, приоритетом ниже жизненно важных работ и явной проверкой эпохи,
исследований и ресурсов. Каменные и более качественные дороги сами не деградируют.

## Производительность и надёжность

- Минимальный вес кэшируется в `NavGrid` и пересчитывается только при изменении
  overrides.
- Для A* используется бинарная очередь; поиск не должен делать линейный обход
  очереди или всей сетки на каждом соседнем переходе.
- Очередь route requests должна иметь бюджет на кадр, когда население вырастет.
  Одинаковые запросы к той же цели можно кэшировать по revision.
- При изменении revision активные и отрицательные маршруты устаревают сразу.
- Сглаживание допускается только по проверенному `segment_cost`; полученные
  waypoints не являются достаточной проверкой безопасности сами по себе.

## Тесты

Минимальный набор автоматических проверок:

- достижимый и недостижимый маршрут;
- запрет диагонального срезания угла;
- выбор более длинного, но дешёвого коридора;
- сохранение корректных сегментов после сглаживания;
- изменение blocked cells и веса увеличивает revision и делает старый
  `RouteResult` устаревшим;
- повторная установка идентичных данных не меняет revision;
- производительный маршрут на максимальном размере игрового поля;
- недостижимый маршрут не выполняется повторно до изменения revision или таймера.
