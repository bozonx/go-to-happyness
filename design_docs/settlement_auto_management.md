# Архитектура авто-управления поселением (Settlement Auto-Management)

## Назначение документа

Этот документ описывает устройство системы автоматического (высокоуровневого) управления поселением. Система отвечает за авто-планирование строительства, автоматическую доставку ресурсов (логистику) и поддержание жизнедеятельности поселения без прямого микроконтроля со стороны игрока. Document описывает интеграцию этой системы с новым ИИ-движком.

## Границы ответственности

Авто-управление отвечает на вопрос: **Какое состояние поселения оптимально, каких ресурсов не хватает и какие глобальные задачи нужно поставить перед жителями?**

### За что отвечает модуль авто-управления (Auto-Management Feature)
- Анализ запасов ресурсов на складах и определение дефицита (например, «не хватает еды в столовой»).
- Планирование цепочек поставок (запросы на логистику).
- Автоматический выбор объектов для строительства или сноса на основе лимитов/правил.
- Принятие решений о включении/выключении рабочих мест в зависимости от сезона или времени суток.

### За что модуль авто-управления НЕ отвечает
- Распределение конкретных жителей на задачи доставки или строительства (это делает `SettlementDirector`).
- Выполнение шагов перемещения ресурсов жителями (это ответственность ИИ-мозга жителя и его актуатора).

## Схема взаимодействия с ИИ

```text
  +-------------------------------------------------------------+
  |                   Settlement Auto-Manager                   |
  |                                                             |
  |  +--------------------+             +--------------------+  |
  |  |  LogisticsManager  |             | ConstructionManager|  |
  |  +---------+----------+             +---------+----------+  |
  |            |                                  |             |
  |            v                                  v             |
  |     [Заявки на сырье]                 [Задачи на стройку]   |
  +------------+----------------------------------+-------------+
               |                                  |
               v                                  v
     +--------------------------------------------------+
     |             SettlementDirector                   |
     |                                                  |
     |  +-------------------+    +-------------------+  |
     |  | LogisticsProvider |    | BuildOrderProvider|  |
     |  +-------------------+    +-------------------+  |
     +--------------------------+-----------------------+
                                |
                                v (публикация приказов)
                          [OrderBoard]
                                |
                                v (выбор ИИ-мозгом жителя)
                          [CitizenBrain]
```

## Ключевые компоненты

### 1. LogisticsManager (Служба авто-логистики)
Анализирует запасы сырья в производственных зданиях и столовых.
- Если в столовой запас еды падает ниже минимального порога, `LogisticsManager` создает логистическую заявку: `(Откуда: Склад, Куда: Столовая, Ресурс: Еда, Количество: X)`.
- Данная заявка передается в `LogisticsProvider` (класс `OrderProvider`), который транслирует её в приказы `CitizenOrder` типа `&"delivery"` для свободных курьеров.

### 2. ConstructionManager (Авто-планировщик строительства)
- Проверяет очередь строительства (например, созданную игроком или автоматическими триггерами вроде «построить дом при нехватке спальных мест»).
- Находит ближайшие ресурсы (дерево, камень) для доставки на стройплощадку.
- Публикует заказы на снабжение стройки и непосредственно на строительство через `BuildOrderProvider`.

## Ключевые инварианты

1. **Разделение фаз чтения-записи**: Авто-планировщики работают на основе снимка мира `WorldSnapshot` и не имеют права напрямую менять состояние сцены Godot в обход транзакций актуаторов.
2. **Отсутствие «мертвых» заказов**: Если склад с ресурсами сгорел или стал недоступен, `LogisticsManager` обязан отменить заявку, а соответствующий `CitizenOrder` должен быть удален из `OrderBoard`. Житель, выполнявший доставку, прерывает задачу по невалидности guard-условия.
3. **Приоритеты авто-задач**: Заказы авто-управления всегда имеют более низкий приоритет, чем прямые указания игрока или критические личные нужды жителей (голод, сон, опасность).

## DX Рекомендации для разработчиков

1. **Изоляция алгоритмов**: Логика авто-управления должна быть чистым кодом без привязки к узлам Godot (3D-нодам). Это позволяет тестировать авто-планирование в Headless-режиме: подать на вход тестовый `WorldSnapshot` с дефицитом еды -> проверить, создались ли нужные заказы на доставку.
2. **Логирование решений**: Каждое решение авто-управления (например, «запущена доставка 5 единиц дерева на стройку №3») должно записывать короткую диагностическую метку в генерируемый `CitizenOrder`. Это позволяет быстро отлаживать поведение ИИ прямо из инспектора жителя.
