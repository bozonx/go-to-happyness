# Архитектура системы приказов (Order & Command System)

## Назначение документа

Этот документ определяет структуру, приоритеты и правила жизненного цикла приказов в игре. Система разделяет непрямые приказы (публикуемые игроком или авто-менеджером на общую доску задач) и прямой ручной контроль над жителями, обеспечивая предсказуемый отклик на действия игрока (Developer & Player Experience).

## Концепция системы приказов

Приказы в игре делятся на три категории по способу выдачи и приоритету:

1.  **Прямые приказы игрока (Direct Manual Control)**: Клик по жителю -> Клик по точке назначения («иди сюда», «атакуй эту цель»). Это синхронный обход ИИ.
2.  **Непрямые приказы игрока (Indirect Player Orders)**: Выделение зоны вырубки, принудительное назначение постройки. Эти задачи попадают на общую доску с наивысшим ИИ-приоритетом.
3.  **Автоматические приказы (Auto-generated Orders)**: Задачи, создаваемые службами логистики, производства и авто-управления.

## Схема прохождения приказов и приоритеты

```text
  [Действие Игрока / Авто-Менеджера]
                  |
                  +--------------------------------+
                  | (Прямой ручной контроль)       | (Непрямой приказ)
                  v                                v
         [Citizen.is_player_controlled = true]     [Поставщик заказов]
                  |                                |
                  | (Синхронный сброс ИИ)          v (Генерация приказа)
                  v                         [CitizenOrder (priority, issuer)]
         [BehaviorRunner.cancel_all()]             |
                  |                                v (Публикация)
                  |                          [OrderBoard]
                  |                                |
                  |                                v (Оценка полезности)
                  |                          [CitizenBrain.think()]
                  |                                |
                  v                                v
         [Ручной Актуатор]                  [ИИ-задача (BehaviorTask)]
```

### Матрица приоритетов ИИ (Utility Priority Matrix)

Для детерминированного выбора между потребностями жителя и глобальными задачами используется следующая шкала полезности (от `0.0` до `1.0`):

| Уровень полезности | Категория | Описание |
| :--- | :--- | :--- |
| **0.90 - 1.00** | Критическое выживание | Критический голод, жажда, угроза жизни, сон при крайнем изнурении. |
| **0.75 - 0.89** | Приказы игрока (Indirect) | Ручные пометки на вырубку, переноску, строительство. |
| **0.50 - 0.74** | Авто-приказы и Расписание | Плановая работа, плановый обед в столовой, сон в положенное время. |
| **0.20 - 0.49** | Фоновые потребности | Легкая усталость, желание отдохнуть в парке, общение. |
| **0.00** | Неприменимо | Цель не имеет условий для выполнения. |

## Мгновенный сброс ИИ при прямом контроле (Direct Control Override)

Когда игрок выделяет жителя и отдает ему прямую команду:
1.  Переменная `is_player_controlled` жителя выставляется в `true`.
2.  Метод [CitizenBrain.think()](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/application/citizen_brain.gd#L25) и `tick()` при первой же проверке видят этот флаг, вызывают `runner.cancel_all(context)` и прекращают выполнение ИИ-ветки.
3.  Житель переходит под полное управление игрока, сбрасывая все удерживаемые ИИ-резервы на ресурсы или рабочие места.
4.  После снятия прямого контроля флаг сбрасывается, и житель возвращается к обычному циклу размышлений (`think_interval`).

## Ключевые инварианты

1.  **Уникальность активного приказа**: На каждый такт размышлений ИИ для одного жителя из [OrderBoard](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/application/order_board.gd) выбирается ровно один приказ с максимальным приоритетом.
2.  **Идемпотентность идентификаторов**: Если содержание приказа (цель, координаты) не изменилось, его `id` сохраняется. Это предотвращает постоянный перезапуск ИИ-задач.
3.  **Привязка к эмитенту**: Поле `issuer` в [CitizenOrder](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/domain/citizen_order.gd) может принимать значения `&"player"` или `&"auto_manager"`. Это позволяет ИИ-мозгу жителя гибко менять формулу полезности в зависимости от того, кто отдал приказ.

## DX Рекомендации

1.  **Безопасный перезапуск шагов**: ИИ-шаги поведения (`BehaviorStep`) должны быть готовы к тому, что приказ изменится или исчезнет в любой момент. Каждая задача, привязанная к приказу, должна реализовывать `guard`-условие проверки актуальности приказа.
2.  **Дебаггер доски приказов**: Для отладки рекомендуется реализовать UI-панель со списком всех текущих заказов на `OrderBoard` с их приоритетами и закрепленными жителями. Это исключит ситуации «почему никто не рубит это дерево».
