# Архитектура системы потребностей юнитов (Unit Needs System)

## Назначение документа

Этот документ описывает архитектуру системы личных потребностей жителей (голод, сон, туалет, гигиена, отдых) и правила их интеграции с системой нативного ИИ. Система отделяет накопление параметров (игровую симуляцию) от принятия решений (ИИ-мозг), обеспечивая высокую расширяемость кода и удобную отладку.

## Границы ответственности

Система потребностей отвечает на вопрос: **Насколько житель голоден или устал, и какие объекты в мире могут эту потребность удовлетворить?**

### За что отвечает модуль потребностей (Needs Feature)
- Симуляция изменения параметров жителей во времени (увеличение голода при активности, падение бодрости).
- Расчет влияния внешних факторов (холод ускоряет трату калорий, работа повышает утомляемость).
- Применение эффектов после удовлетворения потребности (восстановление здоровья после сна, сытость после еды).

### За что модуль потребностей НЕ отвечает
- Выбор пути к кровати или столовой.
- Проверка доступности стола в столовой (это делается через `ReservationLedger`).
- Анимация поедания пищи или сна.

## Схема взаимодействия с ИИ

```text
  +-------------------------------------------------------------+
  |                        Citizen State                        |
  |                                                             |
  |     +------------------+             +------------------+   |
  |     | HungerComponent  |             | Rest/Fatigue Data |   |
  |     +--------+---------+             +--------+---------+   |
  +--------------|--------------------------------|-------------+
                 v (экспорт текущих параметров)   v
     +-----------+--------------------------------+-------------+
     |                 AIWorldFacade.capture()                  |
     |                                                          |
     |   Выгружает факты:                                       |
     |   - "needs.hunger_level": float                          |
     |   - "needs.fatigue_level": float                         |
     +--------------------------+-------------------------------+
                                |
                                v (запись в снимок)
                         [WorldSnapshot]
                                |
           +--------------------+--------------------+
           |                                         |
           v (Оценка полезности)                     v (Оценка полезности)
     [MealGoal.score()]                      [SleepGoal.score()]
           |                                         |
           +--------------------+--------------------+
                                |
                                v (выбор цели с максимальным весом)
                          [UtilityArbiter]
                                |
                                v (выполнение цепочки действий)
                       [BehaviorTask / Steps]
```

## Жизненный цикл потребности и интеграция с ИИ

1.  **Накопление (Tick)**: В основном игровом цикле компоненты потребностей на жителе постепенно меняют свои значения (например, голод увеличивается на 1% за секунду).
2.  **Снимок (Snapshot)**: При вызове [AIWorldFacade.capture()](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/application/settlement_ai_world_facade.gd#L14) текущие уровни потребностей упаковываются в иммутабельные факты [AIFactSet](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/domain/ai_fact_set.gd) конкретного жителя.
3.  **Арбитраж полезности (Arbitration)**:
    - Цель [MealGoal](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/domain/goals/meal_goal.gd) считывает факт `needs.hunger_level`.
    - Функция полезности (Utility Curve) возвращает значение. Например, если голод < 40%, возвращается `0.0`. Если голод > 80%, возвращается экспоненциально растущий вес (вплоть до `0.95`).
    - Если вес потребности превышает вес рабочей задачи, ИИ резервирует личную цель как следующую задачу.
4.  **Завершение текущего шага перед прерыванием (Deferred Preemption)**:
    - При возникновении потребности, превосходящей по приоритету текущую работу, юнит **обязан доделать текущий микро-шаг работы** (например, завершить единичный взмах топором или донести текущую переносимую единицу груза) и только после успешного/неуспешного завершения этого шага переключиться на потребность. Мгновенная отмена посреди шага запрещена (за исключением прямого контроля игрока и синхронной потери доступности).
5.  **Выполнение (Execution)**: Цель строит задачу, шаги которой направляют жителя к столовой, занимают стул (резервирование) и запускают анимацию.
6.  **Применение эффекта (Apply)**: В конце успешного шага ИИ вызывает метод актуатора `apply_food()`, который сбрасывает голод жителя в доменной модели до `0`.

## Ключевые инварианты

1.  **Иммутабельность в цикле принятия решений**: Цели ИИ не меняют значения потребностей напрямую во время оценки полезности. Все изменения параметров происходят только через актуатор в фазе выполнения (`BehaviorStep`).
2.  **Гистерезис переключения**: Чтобы юнит не метался между сном и едой при равных средних значениях, [UtilityArbiter](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/application/utility_arbiter.gd) накладывает дополнительный вес на текущую активную цель (целевой гистерезис).
3.  **Правила бронирования при собирательстве**:
    - При прерывании работы по сбору ресурсов (рубка дерева, сбор веток/травы/пищи в лесу) бронь на ресурс в [ReservationLedger](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/domain/reservation_ledger.gd) **не сохраняется** и принудительно сбрасывается.
    - При возобновлении или повторном старте сбора выбор цели идет среди свободных (не занятых другими жителями) ресурсов. Предпочтение отдается **ближайшим** ресурсам, а среди равных — тем, у которых **осталось немного ресурса до полного сбора** (чтобы быстрее закончить очистку площадки).
4.  **Защита от застревания (Cooldown)**: Если шаг удовлетворения потребности провалился (например, еда в столовой кончилась), цель получает `FAILURE_COOLDOWN` на переоценку, предотвращая зацикливание ИИ на невыполнимой потребности, если есть другие доступные цели.

## DX Рекомендации для разработчиков

1.  **Масштабируемость полезности**: Используйте математические кривые (например, `y = x^2` или кривую Безье) для расчета полезности потребностей, чтобы избежать резких скачков поведения юнитов.
2.  **Тестирование**: Создавайте тесты домена, симулируя критические уровни потребностей на снимке и проверяя, что [UtilityArbiter](file:///home/ivank/_personal/codeWorkspace/go-to-happyness/game/features/decision/application/utility_arbiter.gd) корректно выбирает нужную цель вместо фоновой работы.

## Текущее распределение сценариев

Все реализованные личные сценарии жителя проходят через нативный ИИ и имеют
ровно одного владельца решения:

| Сценарий | Владелец состояния и расписания | Goal / Step | Команда актуатора |
| --- | --- | --- | --- |
| Прием пищи | `CanteenService` | `MealGoal` / `EatAtCanteenStep` | `eat` |
| Ночной сон | расписание смены в `SimulationDayCycle` | `SleepGoal` / `SleepAtHomeStep` | `sleep` |
| Физиологическая нужда | `CitizenNeedsService` | `ToiletGoal` / `RelieveStep` | `relieve` |
| Плановый и фоновый отдых | `CitizenNeedsService` | `RestGoal` / `MoveToStep` + `RelaxAtPositionStep` | `move_to`, `relax` |

`CitizenActor` исполняет только уже выбранную команду: движение, ожидание слота,
анимацию и сигнал завершения. Он не хранит расписание туалета, не выбирает туалет
или куст и не отправляет себя в парк. Для физиологической нужды `RelieveStep`
резервирует конкретный слот в `ReservationLedger`; при отмене, неудаче или
завершении резерв освобождается.

Гигиена пока не имеет игровой механики в кодовой базе.
Её нельзя добавлять как фиктивную goal: сначала нужна вертикаль с доменным
параметром, фактами фасада, объектом удовлетворения, актуатором и тестами по
правилу добавления механики выше.
