# Модульный редактор зданий

## 1. Назначение и границы миграции

Модульный редактор создаёт единый открытый формат `.gdbuilding.json` для
встроенных и пользовательских зданий. Старые здания продолжают использовать
`building_blueprints.gd`; новый resolver выбирает JSON только для уже созданных
модульных определений. Массовая миграция старых зданий выполняется отдельно.

Здание разделяется на три понятия:

- `BuildingBlueprint` — неизменяемое описание содержимого;
- `BuildingRuntimeState` — состояние построенного экземпляра и активных зон;
- `BuildingView` — меши, коллизии и маркеры Godot.

Редактор находится в feature `buildings`:

```text
game/features/buildings/
  domain/editor/       # формат, блоки, материалы, зоны, сетка
  application/         # состояние экземпляра и runtime зон
  presentation/        # файловый resolver и игровые визуальные адаптеры
  presentation/editor/ # сцена редактора, repository и визуальные инструменты
  data/blueprints/     # встроенные файлы
```

## 2. Сетка и каркас

Основная строительная сетка имеет ячейку `1 × 1 × 1 м` и целочисленные
координаты `Vector3i`. Она используется для кубов, плит, панелей, проёмов,
углов, крыш, лестниц и ограждений.

Шаг размещения мебели и декора не фиксируется форматом каркаса. Для отдельных
объектов будущий режим декора может использовать настраиваемый snap либо
свободное размещение. Значение `0.25 м` не является обязательным инвариантом
формата.

Каждый блок содержит:

```json
{
  "pos": [0, 0, 0],
  "block_id": "wall_panel",
  "material_id": "earth",
  "rot": 0
}
```

`block_id` определяет геометрию, `material_id` — внешний материал каркаса и
реальный строительный ресурс.

Доступные материалы:

| material_id | Игровой ресурс | Минимальная эра |
|---|---|---|
| `branches` | `branches` | tent |
| `earth` | `soil` | earth |
| `clay` | `clay` | clay |
| `wood` | `boards` | wood |
| `stone` | `stone` | stone |
| `brick` | `bricks` | brick |

Стоимость `construction_cost` вычисляется из блоков при сохранении. Редактор не
создаёт новые идентификаторы ресурсов, отсутствующие в `ResourceIds`.

## 3. Режимы редактора

Редактор имеет четыре независимых этапа.

### 3.1. Каркас

- установка и удаление блоков;
- поворот на четверть оборота;
- выбор уровня Y;
- линия и в будущем прямоугольная кисть;
- выбор материала каждого устанавливаемого блока;
- surface/underground construction style.

Подземный стиль в формате зарезервирован, но terrain-cut и автоматический
фундамент требуют отдельного вертикального среза.

### 3.2. Отделка

Отделка изменяет поверхности уже установленных блоков: штукатурку, обои,
фасады, плинтусы, карнизы и другие покрытия. Она хранится отдельно в
`surface_finishes` и `decor_trims` и не меняет ресурс каркасного блока.

### 3.3. Декор и наполнение

Этот режим размещает мебель, оборудование, свет и декоративные объекты. Они
хранятся в `objects`. Объект может иметь собственный шаг привязки и свободный
поворот. Функциональные рабочие точки не следует кодировать только мебелью:
они принадлежат активной зоне.

### 3.4. Активные зоны

Зона имеет устойчивый `zone_id`, назначение, профессию, предел работников,
занимаемые ячейки, рабочие якоря и локальные поддоны.

```json
{
  "id": "bakery_1",
  "name": "Пекарня",
  "kind": "workplace",
  "profession_type": "cook",
  "max_workers": 2,
  "cells": [[1, 0, 1], [2, 0, 1]],
  "work_anchors": [
    {"id": "oven", "pos": [1.5, 0.0, 1.5], "rot": [0, 0, 0], "action": "work"}
  ],
  "storage_trays": {
    "input": {"pos": [0.5, 0.0, 1.5], "capacity": 50},
    "output": {"pos": [3.5, 0.0, 1.5], "capacity": 50}
  }
}
```

Поддерживаемые `kind`: `workplace`, `civic`, `trade`, `storage`, `housing`,
`leisure`.

## 4. Runtime активных зон и AI

При завершении модульного здания определения зон преобразуются в типизированные
`ActiveWorkZoneState` внутри `BuildingRuntimeState`.

- назначение хранит стабильные `ai_id`, а не `ObjectID`;
- `BuildingZoneService` распределяет жителей их рабочим зонам;
- AI читает мировой рабочий якорь через существующий `AIWorldFacade`;
- движение и действие выполняются только через `CitizenActuator`;
- старые здания без зон продолжают использовать старые service positions и
  списки `building_type`.

Первый игровой срез публикует рабочие места для уже совместимых service-ролей:
`cook`, `teacher`, `seller`, `craftsman`. Производственные рецепты, специализированные
действия поддонов и новые профессии добавляются отдельными вертикальными
срезами: facts → provider → goal → steps → actuator → reservations → tests.

## 5. Встроенные и пользовательские файлы

Один формат используется двумя источниками:

```text
builtin: res://game/features/buildings/data/blueprints/
player:  user://custom_buildings/
```

Runtime-ключ встроенного здания равен его ID. Пользовательский ключ имеет
namespace `user:<id>`, поэтому пользовательский файл не может незаметно
переопределить стандартное здание.

Player mode позволяет задать ID, имя, эру, footprint и стандартное fallback
здание. Пользовательские файлы перечисляются отдельной категорией меню
строительства. Dev mode сохраняет тот же формат во встроенную папку и позднее
получит инструменты компиляции меша и NavMesh-preview.

Сохранение Blueprint атомарно: сначала пишется `.tmp`, затем файл заменяется.
Перед записью проверяются версия, ID, размеры, блоки, материалы, эра,
повторяющиеся координаты и ID зон.

## 6. Ссылка из игрового сейва и fallback

Пользовательский Blueprint не встраивается целиком в обычный сейв. Постройка
сохраняет ссылку:

```json
{
  "blueprint_ref": {
    "source": "player",
    "id": "my_bakery",
    "revision": "7fa2d194",
    "fallback_building_id": "house"
  }
}
```

Рядом сохраняется только изменяемое runtime-состояние зон:

- существующие zone IDs и определения, необходимые для восстановления;
- назначенные `ai_id`;
- локальные якоря и поддоны;
- ссылка рабочего места жителя на стабильную клетку здания.

Алгоритм загрузки:

1. Найти файл по `(source, id)`.
2. Если файл найден, загрузить актуальную геометрию и наложить сохранённое
   runtime-состояние зон.
3. Если файл отсутствует, построить `fallback_building_id`.
4. Наложить сохранённые зоны на fallback, чтобы рабочие назначения и якоря не
   исчезли полностью.
5. Если отсутствует и fallback, пропустить объект с диагностикой.

`revision` диагностирует изменение файла. Строгое закрепление старой геометрии
не требуется: пользовательские здания остаются самостоятельными редактируемыми
файлами.

## 7. Формат `.gdbuilding.json`, версия 1

```json
{
  "version": 1,
  "id": "earth_workshop",
  "name": "Земляная мастерская",
  "construction_style": "surface",
  "category": "earth",
  "fallback_building_id": "earth_house",
  "grid_bounds": {"x": 6, "y": 3, "z": 5},
  "pivot_offset": {"x": 3, "y": 0, "z": 2},
  "footprint": [6, 5],
  "entrance": [0, -2],
  "worker_entrances": [[0, -2]],
  "blocks": [],
  "surface_finishes": [],
  "decor_trims": [],
  "objects": [],
  "work_zones": [],
  "construction_cost": {"soil": 20}
}
```

Версия, отличная от поддерживаемой, отклоняется. Мигратор пока не нужен:
изменение версии допускается только вместе с явной задачей миграции.

## 8. Производительность

Текущий переходный renderer создаёт модуль на блок и подходит для пилотных
Blueprint. До массовой миграции старых и крупных зданий необходим compiler:

```text
Blueprint
  → удаление внутренних граней
  → batching по материалам
  → объединённые коллизии
  → LOD
  → игровой BuildingView
```

Отсутствие compiler не блокирует подготовку формата, зон и пользовательского
каталога, но блокирует массовый перевод сложных зданий.
