# Модульный редактор зданий

## 1. Назначение и границы миграции

Модульный редактор создаёт единый открытый формат `.gdbuilding.json` для
встроенных и пользовательских зданий. Старые здания продолжают использовать
`building_blueprints.gd`; новый resolver выбирает JSON только для уже созданных
модульных определений. Массовая миграция старых зданий выполняется отдельно.

Здание разделяется на три понятия:

- `BuildingBlueprint` — неизменяемое описание содержимого;
- `BuildingRuntimeState` — состояние построенного экземпляра и активных зон;
- `BuildingView` — меши, коллизии и маркеры Godot.

Редактор находится в feature `buildings`:

```text
game/features/buildings/
  domain/editor/       # формат, блоки, материалы, зоны, сетка
  application/         # состояние экземпляра и runtime зон
  presentation/        # файловый resolver и игровые визуальные адаптеры
  presentation/editor/ # сцена редактора, repository и визуальные инструменты
  data/blueprints/     # встроенные файлы
```

## 2. Сетка и каркас

Основная строительная сетка имеет ячейку `1 × 1 × 1 м` и целочисленные
координаты `Vector3i`. Она используется для кубов, плит, панелей, проёмов,
углов, крыш, лестниц и ограждений.

Шаг размещения мебели и декора не фиксируется форматом каркаса. Для отдельных
объектов будущий режим декора может использовать настраиваемый snap либо
свободное размещение. Значение `0.25 м` не является обязательным инвариантом
формата.

Каждый блок содержит:

```json
{
  "pos": [0, 0, 0],
  "block_id": "wall_panel",
  "material_id": "earth",
  "rot": 0
}
```

`block_id` определяет геометрию, `material_id` — внешний материал каркаса и
реальный строительный ресурс.

Материал не выбирается напрямую из полного каталога. Сначала выбирается **эра
здания** (`category`), и уже она задаёт набор доступных материалов. Материалы
кумулятивны: здание эры может использовать материалы своей эры и всех
предыдущих, поэтому в каждой эре доступно несколько материалов. Материал по
умолчанию — «определяющий» материал самой эры (например `earth` для земляной).

| material_id | Игровой ресурс | Эра появления |
|---|---|---|
| `branches` | `branches` | tent |
| `thatch` | `grass` | tent |
| `tarp` | `tarp` | tent |
| `earth` | `soil` | earth |
| `clay` | `clay` | clay |
| `logs` | `logs` | wood |
| `wood` | `boards` | wood |
| `stone` | `stone` | stone |
| `brick` | `bricks` | brick |

Порядок эр: `tent → earth → clay → wood → stone → brick`. Сохранение
проверяет, что каждый блок использует материал, доступный в выбранной эре;
блок с материалом более поздней эры отклоняется. Список материалов в редакторе
перестраивается при смене эры, а редактор предупреждает, если уже размещённые
блоки не соответствуют новой эре.

Стоимость `construction_cost` вычисляется из блоков при сохранении. Редактор не
создаёт новые идентификаторы ресурсов, отсутствующие в `ResourceIds`.

## 3. Режимы редактора

Редактор имеет четыре независимых этапа.

### 3.1. Каркас

- установка и удаление блоков;
- поворот на четверть оборота;
- выбор уровня Y;
- две кисти: **линия** (протягивание) и **прямоугольник** (заливка целого пола
  или потолка на активном слое Y). Прямоугольная кисть работает и для установки,
  и для стирания;
- материал берётся из выбранной эры (см. §2), а не выбирается для каждого блока
  из полного каталога;
- стиль постройки `surface` / `underground`.

Подземные сооружения (`underground`) доступны только начиная с земляной эры
(`earth`) — раньше нет ни инструментов, ни технологии рытья. В редакторе пункт
«Подземная» заблокирован для палаточной эры, а сохранение отклоняет подземное
здание эры `tent`.

**Типы блоков.** Помимо конструкции, стен, крыш, проходов и ограждений
существует отдельный тип **фундамент** (`foundation`). Его особенность —
поведение при рендере: если здание стоит на неровной поверхности, фундаментные
блоки автоматически продолжаются вниз до земли, чтобы под домом не оставалось
пустоты. В формате это обычный блок; «уход под землю» вычисляется презентацией
по рельефу, а не хранится покоординатно.

### 3.2. Отделка

Отделка изменяет поверхности уже установленных блоков: штукатурку, обои,
фасады, плинтусы, карнизы и другие покрытия. Она хранится отдельно в
`surface_finishes` и `decor_trims` и не меняет ресурс каркасного блока.

### 3.3. Декор и наполнение

Этот режим размещает мебель, оборудование, свет и декоративные объекты. Они
хранятся в `objects`. Объект может иметь собственный шаг привязки и свободный
поворот. Функциональные рабочие точки не следует кодировать только мебелью:
они принадлежат активной зоне.

### 3.4. Активные зоны

Активные зоны делятся на **три уровня**. Первый — самостоятельная сущность
«место»; второй и третий — **якоря** с общей структурой (`pos`/`rot`/`role`/
`owner`), различаемые полем `role`. В редакторе всё это — один режим «Зоны», но
уровни авторятся раздельно.

Данные хранятся в двух массивах: `place_zones[]` (уровень 1) и `zone_anchors[]`
(уровни 2 и 3). Якорь ссылается на своё место через `owner`; `owner` может быть
пустым — тогда это мировой якорь (например автобусная остановка на улице, не
принадлежащая зданию).

#### Уровень 1 — зона места (`place_zones`)

«Что это за место»: целый дом, комната-склад или комната-магазин в торговом
центре, жилая ячейка, зона отдыха. Игрок кликает именно по зоне места — так
открывается меню здания. Здесь же живёт занятость: профессия и предел
работников. **Где** именно стоят работники — задаётся слотами уровня 2; если
слотов нет, юниты занимают места случайно внутри `cells`.

```json
{
  "id": "bakery_1",
  "name": "Пекарня",
  "kind": "workplace",
  "subtype": "",
  "profession_type": "cook",
  "max_workers": 2,
  "cells": [[1, 0, 1], [2, 0, 1]]
}
```

Поддерживаемые `kind`: `workplace`, `civic`, `trade`, `storage`, `housing`,
`leisure`, `special`. Занятость дают только `workplace`/`civic`/`trade`. Крупное
здание (ТЦ, блок бункера) содержит несколько зон места — по одной на комнату.

Зоны места различаются подтипом `subtype`:

- **`leisure`** — рекреация; `subtype`: `park`, `plaza`, `playground`,
  `sports_field`, `gym`, `cinema`, `theater`. Подтип определяет, какую
  потребность в отдыхе закрывает посещение.
- **`special`** — особые немонетизируемые маркеры; `subtype`: `entrance_sign`
  (въездной знак поселения), `monument`, `notice_board`, `flagpole`. Геометрия
  свободна: игрок может пометить хоть триумфальную арку как `entrance_sign`.

Поле `subtype` пусто для «плоских» `kind` (workplace, civic, trade, storage,
housing). В редакторе список подтипов появляется только для `leisure` и
`special`.

#### Уровень 2 — слоты (`zone_anchors`, роли-слоты)

Слот — точка, которую юнит **занимает** для действия: место за прилавком,
ресепшен, офисный стол, кровать для сна, точка рыбалки у пруда, поддоны склада.
Смысл преимущественно декоративный/размещающий — чтобы человечки занимали
конкретные места, а не стояли в центре. Если у зоны места слотов нет, точки
занятости выбираются случайно внутри её `cells`.

Роли-слоты: `work`, `counter`, `reception`, `desk`, `bed`, `rest`, `fishing`,
`input_tray`, `output_tray`. Поддоны склада — это тоже слоты (`input_tray`/
`output_tray`), у которых `capacity` означает ёмкость хранения, а не число
занимающих.

#### Уровень 3 — зоны маршрутизации (`zone_anchors`, роли-маршрутизации)

Точки, которые юнит **проходит или у которых ждёт**, а не занимает: двери для
посетителей и служебные двери, остановки такси и автобуса. Используются для
маршрутизации, а не для занятости. Могут не принадлежать зданию (`owner` пуст) —
например остановка посреди города.

Роли-маршрутизации: `visitor_door`, `service_door`, `taxi_stop`, `bus_stop`.
Двери-маршрутизации — авторитетное представление входов; на уровне размещения на
доске остаются `entrance`/`worker_entrances` (§7) как проекция дверей, пока их не
начнёт напрямую потреблять навигация.

```json
"zone_anchors": [
  {"id": "counter", "owner": "shop_1",  "role": "counter",      "pos": [1.5, 0.0, 1.5], "rot": [0, 0, 0],   "capacity": 1},
  {"id": "in_1",    "owner": "shop_1",  "role": "input_tray",   "pos": [0.5, 0.0, 1.5], "rot": [0, 0, 0],   "capacity": 50},
  {"id": "vdoor",   "owner": "shop_1",  "role": "visitor_door", "pos": [1.5, 0.0, 0.0], "rot": [0, 180, 0], "capacity": 1},
  {"id": "bus_a",   "owner": "",        "role": "bus_stop",     "pos": [8.0, 0.0, -2.0],"rot": [0, 90, 0],  "capacity": 1}
]
```

## 4. Runtime активных зон и AI

При завершении модульного здания авторские `place_zones` + `zone_anchors`
**денормализуются** на границе (`building_blueprints`) обратно в один список зон
на здание: якоря группируются по своему `owner`, слоты становятся рабочими
якорями зоны места, поддоны — её поддонами, а зоны маршрутизации выносятся
отдельным списком `routing_anchors`. Этот денормализованный список рантайм
принимает без изменений как типизированные `ActiveWorkZoneState` внутри
`BuildingRuntimeState`.

- занятость и вместимость берутся из зоны места (`profession`/`max_workers`);
- рабочая точка жителя — слот его зоны места; если слотов нет, берётся центр
  `cells` зоны;
- назначение хранит стабильные `ai_id`, а не `ObjectID`;
- `BuildingZoneService` распределяет жителей их зонам места;
- AI читает мировой рабочий якорь через существующий `AIWorldFacade`;
- зоны маршрутизации (двери, остановки) сохраняются как `routing_anchors` и
  предназначены для навигации; занятость их не потребляет;
- движение и действие выполняются только через `CitizenActuator`;
- старые здания без зон продолжают использовать старые service positions и
  списки `building_type`.

Первый игровой срез публикует рабочие места для уже совместимых service-ролей:
`cook`, `teacher`, `seller`, `craftsman`. Производственные рецепты, специализированные
действия поддонов и новые профессии добавляются отдельными вертикальными
срезами: facts → provider → goal → steps → actuator → reservations → tests.

## 5. Встроенные и пользовательские файлы

Один формат используется двумя источниками:

```text
builtin: res://game/features/buildings/data/blueprints/
player:  user://custom_buildings/
```

Runtime-ключ встроенного здания равен его ID. Пользовательский ключ имеет
namespace `user:<id>`, поэтому пользовательский файл не может незаметно
переопределить стандартное здание.

Player mode позволяет задать ID, имя, эру, footprint и стандартное fallback
здание. Пользовательские файлы перечисляются отдельной категорией меню
строительства.

Dev mode — это авторский инструмент для **стандартных зданий игры**, а не
отдельная песочница. Он сохраняет в каноническую папку
`res://game/features/buildings/data/blueprints/`, из которой читает игровой
`BuildingBlueprintLibrary`, поэтому созданное и отредактированное в dev-режиме
здание — ровно то, что попадает в игру как стандартное. Dev mode позднее
получит инструменты компиляции меша и NavMesh-preview.

Сохранение Blueprint атомарно: сначала пишется `.tmp`, затем файл заменяется.
Перед записью проверяются версия, ID, размеры, блоки, материалы, эра,
соответствие материала блока эре здания, допустимость подземного стиля для
эры, повторяющиеся координаты и ID зон.

## 6. Ссылка из игрового сейва и fallback

Пользовательский Blueprint не встраивается целиком в обычный сейв. Постройка
сохраняет ссылку:

```json
{
  "blueprint_ref": {
    "source": "player",
    "id": "my_bakery",
    "revision": "7fa2d194",
    "fallback_building_id": "house"
  }
}
```

Рядом сохраняется только изменяемое runtime-состояние зон:

- существующие zone IDs и определения, необходимые для восстановления;
- назначенные `ai_id`;
- локальные якоря и поддоны;
- ссылка рабочего места жителя на стабильную клетку здания.

Алгоритм загрузки:

1. Найти файл по `(source, id)`.
2. Если файл найден, загрузить актуальную геометрию и наложить сохранённое
   runtime-состояние зон.
3. Если файл отсутствует, построить `fallback_building_id`.
4. Наложить сохранённые зоны на fallback, чтобы рабочие назначения и якоря не
   исчезли полностью.
5. Если отсутствует и fallback, пропустить объект с диагностикой.

`revision` диагностирует изменение файла. Строгое закрепление старой геометрии
не требуется: пользовательские здания остаются самостоятельными редактируемыми
файлами.

## 7. Формат `.gdbuilding.json`, версия 1

```json
{
  "version": 1,
  "id": "earth_workshop",
  "name": "Земляная мастерская",
  "construction_style": "surface",
  "category": "earth",
  "fallback_building_id": "earth_house",
  "grid_bounds": {"x": 6, "y": 3, "z": 5},
  "pivot_offset": {"x": 3, "y": 0, "z": 2},
  "footprint": [6, 5],
  "entrance": [0, -2],
  "worker_entrances": [[0, -2]],
  "blocks": [],
  "surface_finishes": [],
  "decor_trims": [],
  "objects": [],
  "place_zones": [],
  "zone_anchors": [],
  "construction_cost": {"soil": 20}
}
```

`entrance`/`worker_entrances` — проекция дверей-маршрутизации на клетки доски
размещения (координатное пространство footprint, а не авторская 3D-сетка). Пока
навигация не потребляет якоря напрямую, эти поля остаются и задаются автором.

Версия остаётся `1`. Файлы прошлого поколения с массивом `work_zones` читаются
через встроенную миграцию на загрузке: каждая старая зона разбивается на одну
`place_zones`-запись и её `zone_anchors` (рабочие якоря → слоты, поддоны →
`input_tray`/`output_tray`). Запись всегда выполняется в новом формате.

## 8. Производительность

Текущий переходный renderer создаёт модуль на блок и подходит для пилотных
Blueprint. До массовой миграции старых и крупных зданий необходим compiler:

```text
Blueprint
  → удаление внутренних граней
  → batching по материалам
  → объединённые коллизии
  → LOD
  → игровой BuildingView
```

Отсутствие compiler не блокирует подготовку формата, зон и пользовательского
каталога, но блокирует массовый перевод сложных зданий.
