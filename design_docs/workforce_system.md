# Архитектура системы работ и занятости (Workforce System)

## Назначение документа

Этот документ описывает архитектуру системы работ, управления вакансиями, специализациями и занятостью жителей. Он определяет границы ответственности между игровыми механиками (производство, услуги) и системой нативного ИИ, а также задает правила интеграции новых профессий.

## Границы ответственности

Система занятости отвечает на вопрос: **Кто где работает, какие рабочие места свободны и какие требования предъявляются к кандидатам?**

### За что отвечает модуль занятости (Workforce Feature)
- Учет свободных и занятых рабочих мест (`Workplace`).
- Управление профилем навыков и специализацией жителей.
- Назначение жителей на вакансии (постоянные и временные).
- Увольнение жителей при разрушении зданий или смене приоритетов игроком.

### За что модуль занятости НЕ отвечает
- Управление перемещением жителей к месту работы.
- Проигрывание анимаций работы и логика добычи ресурсов в процессе.
- Принятие решений о том, идти ли жителю работать прямо сейчас (это ответственность `CitizenBrain` и системы потребностей).

## Общая схема интеграции с ИИ

```text
  +-------------------------------------------------------------+
  |                        Workforce Module                     |
  |                                                             |
  |  +------------------+             +----------------------+  |
  |  | WorkplaceRegistry|             | WorkforceCoordinator |  |
  |  +--------+---------+             +----------+-----------+  |
  |           |                                  |              |
  |           v                                  v              |
  |   [Active Workplaces]                 [Citizen Skills]      |
  +-----------+----------------------------------+--------------+
              |                                  |
              v                                  v
     +------------------+               +------------------+
     |   Workplace      |               |  CitizenActor    |
     |   (Production)   |               |  (Employment)    |
     +--------+---------+               +------------------+
              |
              v (публикует вакансию)
     +----------------------------------+
     |      WorkforceOrderProvider      |
     +----------------+-----------------+
                      |
                      v (генерирует)
               [CitizenOrder]
                      |
                      v (записывает)
                [OrderBoard] <------+ (выбирает лучший приказ)
                                    |
                            [CitizenBrain] (ИИ жителя)
                                    |
                                    v (если выбран приказ работы)
                             [WorkGoal.score()]
                                    |
                                    v
                            [WorkBehaviorTask]
                                    |
                                    v (вызывает способности)
                            [CitizenActuator]
```

## Компоненты системы

### 1. Workplace (Рабочее место)
Компонент или интерфейс, прикрепленный к зданиям (пилорама, ферма, столовая).
- Хранит максимальное число сотрудников и список текущих работников (`ai_id`).
- Определяет тип требуемой роли (например, `"woodcutter"`, `"cook"`).
- Сообщает о своей доступности для работы (например, пилорама не работает, если склад забит деревом).

### 2. WorkforceOrderProvider
Класс, реализующий интерфейс `OrderProvider` для ИИ-системы.
- Периодически сканирует зарегистрированные `Workplace`.
- Для каждого прикрепленного сотрудника, у которого наступило рабочее время, генерирует персональный `CitizenOrder` с типом работы (например, `kind = &"forestry"`) и приоритетом, зависящим от важности здания.
- Для свободных вакансий отправляет запросы в `WorkforceCoordinator` для назначения свободных резервных жителей.

### 3. WorkGoal (ИИ-цель работы)
Реализует интерфейс `AICitizenGoal` в модуле принятия решений.
- **`score()`**: Оценивает приказ работы. Если приказ активен, возвращает базовую полезность работы (например, `0.5`). Если у жителя высокий уровень усталости или голода, полезность личных нужд перевесит полезность работы.
- **`build_task()`**: Создает дерево шагов для работы. Например, для пилорамы: `Sequence[MoveTo(Workplace), WorkAtBuilding(Duration), DeliverProductToWarehouse]`.

## Ключевые инварианты

1. **Закрепление постоянной работы и жилья (Assignment Persistence)**:
   - Место постоянной работы (`permanent_workplace`) и жилье (`home`) остаются жестко закрепленными за жителем до тех пор, пока:
     - Игрок не вмешается вручную (уволит/выселит через интерфейс).
     - Рабочее здание или дом не будут разрушены/демонтированы.
     - Не пропадет работа в течение длительного периода (закрытие вакансии, прекращение деятельности).
   - Временное отсутствие жителя на рабочем месте (сон, еда, лечение) **не является** причиной для автоматического увольнения или выселения.
2. **Регистрация безработицы (Unemployment Flow)**:
   - Если житель теряет работу (разрушение здания, закрытие вакансий) и `WorkforceCoordinator` не может подобрать ему новое постоянное место в течение лимита времени, ИИ жителя активирует специальный сценарий.
   - Житель получает автоматический приказ `kind = &"register_unemployment"`. Его цель строит задачу, направляющую жителя в Центр Занятости (`Employment Center`) для официального оформления статуса безработного и поиска свободных вакансий резерва.
3. **Единственность занятости**: Житель (`ai_id`) может быть трудоустроен (иметь `permanent_workplace`) только в одном месте одновременно.
4. **Детерминизм назначения**: Назначение на вакансии происходит централизованно через `WorkforceCoordinator` во время тика директора, исключая ситуацию, когда два жителя одновременно занимают одно место.
5. **Авто-увольнение**: При уничтожении здания, отзыве вакансий игроком или смерти жителя связь `Workplace <-> Citizen` разрывается синхронно, а все активные резервы на ресурсы освобождаются.
6. **Смена смен**: Время работы определяется глобальным расписанием поселения. Если рабочая смена закончилась, `WorkforceOrderProvider` прекращает публиковать приказ работы или снижает его приоритет до `0.0`.

## Правила добавления новой профессии (DX)

Чтобы добавить в игру новую профессию (например, Шахтер):
1. В кодовой базе шахты (модуль `game/features/production`) создайте или настройте компонент `Workplace` с ролью `"miner"`.
2. Зарегистрируйте шахту в системе занятости, чтобы `WorkforceCoordinator` знал о вакансиях шахтеров.
3. Добавьте в `WorkforceOrderProvider` обработку роли `"miner"` (генерация приказа работы на шахте).
4. Напишите `MinerGoal` (или расширьте универсальный `WorkGoal`) и соответствующие шаги поведения (`MoveToMineStep`, `MineOreStep`).
5. Напишите тесты на выполнение контракта: юнит получает приказ -> идет в шахту -> добывает руду -> при достижении усталости прерывается на сон.

## Текущее состояние переноса

Рабочие вертикальные срезы реализованы для постоянных лесоруба и фермера:

- `ForestryOrderProvider` публикует один стабильный приказ `forestry` на
  доступного постоянного работника;
- provider выбирает различающиеся деревья из снимка, а `ForestryWorkStep`
  резервирует конкретное дерево в `ReservationLedger` перед запуском;
- `SettlementCitizenActuator` вызывает существующее исполнение цикла
  «дерево -> пилорама», но его запуск и следующий цикл принадлежат native AI;
- `CitizenGoapBrain` больше не выдает работу постоянному лесорубу.
- `FarmingOrderProvider` публикует один приказ `farming` для постоянного
  работника фермы; `FarmingWorkStep` выполняет производство, а курьер забирает
  готовую еду в рамках существующей логистической задачи.
- `CitizenGoapBrain` и прямые вызовы `WorkforceCoordinator.assign_work()` больше
  не выдают работу постоянному фермеру.

Автоматическое назначение на постоянную вакансию пока остается в
`WorkforceCoordinator`. Следующие срезы должны перенести рабочие роли по одной:
строительство, сбор, производство и логистика.
