# Дизайн-документ: навигация и дороги

## Назначение

Система определяет, по каким клеткам могут передвигаться жители и транспорт, как
выбирается маршрут и как покрытие меняет его стоимость и фактическую скорость.
Дороги являются данными покрытия, а не вторым маршрутизатором: маршрут всегда
строится только через `NavGrid` и `GridRouteService`.

Цели системы:

- жители не проходят через здания, воду и опасные участки;
- более дешёвое покрытие естественно притягивает маршруты;
- изменение препятствий немедленно делает активные маршруты небезопасными, а
  изменение покрытия влияет на выбор следующих маршрутов;
- работа остаётся предсказуемой при большом числе жителей.

## Текущее состояние без дорог

Уже реализованы следующие компоненты:

- `NavGrid` хранит размеры клеток, блокировки, общие разреженные
  переопределения веса и profile-specific переопределения; клетка без
  переопределения имеет вес травы `2.0`.
- `RouteRequest` передаёт `from`, `destination`, `traveler_profile` и
  `allow_destination_cell`. Старый `find_route(from, destination)` сохранён как
  pedestrian-wrapper.
- `GridRouteService` использует взвешенный A* по восьми направлениям. Диагональ
  имеет длину `sqrt(2)` и запрещена, если хотя бы одна из прилегающих
  ортогональных клеток заблокирована.
- A* использует бинарную очередь приоритетов и допустимую octile-эвристику,
  умноженную на минимальный вес сетки.
- После поиска weighted string pulling удаляет изгибы, только если прямой отрезок
  не пересекает препятствия и стоит не более чем на 8% дороже заменяемого участка.
- `NavGrid.segment_cost()` использует Amanatides & Woo и проверяет обе клетки при
  пересечении угла, поэтому путь не срезает угол здания.
- Скорость пешехода равна `base_speed / cell_weight`; трава с весом `2.0` даёт
  50% базовой скорости.
- `TrailFieldService` хранит непрерывный визуальный след и отдельную разреженную
  nav-проекцию тропинок. При входе жителя в новую nav-клетку сервис накапливает
  силу, переводит клетку в young/mature/degrading состояния и применяет веса
  только к pedestrian-профилю через `NavGrid.set_profile_cell_weights()`.

`NavGrid` владеет монотонными `revision` и `topology_revision`. `revision` —
ревизия выбора маршрута: она увеличивается при любом реальном изменении размеров,
блокировок или весов. `topology_revision` — ревизия безопасности: она изменяется
только когда меняется проходимость. `RouteResult` содержит обе ревизии. Актор
обязан перепланировать путь при несовпадении `topology_revision`; смена веса
покрытия меняет выбор новых маршрутов и скорость, но не отменяет безопасный уже
построенный путь. `RouteResult.is_topologically_stale()` является единым API этой
проверки.

Недостижимый маршрут повторяется с экспоненциальной задержкой от 2 до 16 секунд.
Изменение revision отменяет ожидание и разрешает немедленную попытку.

## Модель стоимости

Для перехода из клетки `A` в соседнюю клетку `B`:

```text
cost(A -> B) = geometric_distance(A, B) * traversal_weight(B, profile)
```

Вес всегда положителен. `1.0` является базовой скоростью пешего движения;
значения меньше `1.0` допустимы для улучшенных дорог. Блокировка не выражается
весом: такая клетка вообще исключается из поиска.

Веса тропинок намеренно близки к базовому весу травы (`2.0`): молодая тропинка
имеет вес `1.7`, полноценная — `1.4`. Это даёт умеренное ускорение (~1.4x для
зрелой тропинки), но не делает его двукратным. Двукратное ускорение (`вес 1.0`)
расширило бы каждый маршрут: A*-эвристика умножается на минимальный вес сетки,
и снижение веса даже нескольких клеток увеличивает эвристическую оценку для
всех маршрутов. Компромисс сохраняет tight heuristic и приемлемую стоимость
поиска по мере роста сети тропинок.

Базовые значения для будущего покрытия:

| Покрытие | Вес | Скорость пешехода |
| --- | ---: | ---: |
| Трава | 2.0 | 50% |
| Песок, кустарник | 3.0–4.0 | 25–33% |
| Болото, густой лес | 8.0 или запрет | 12.5% |
| Полноценная тропинка палаточной эры | 1.4 | 71% |
| Грунтовая дорога | 1.0 | 100% |
| Глинобитная | 0.9 | 111% |
| Деревянная | 0.85 | 118% |
| Каменная | 0.8 | 125% |
| Асфальтовая | 0.7 | 143% |
| Асфальто-бетонная | 0.6 | 167% |

Игровые коэффициенты из таблицы являются начальными значениями баланса. Скорость
и стоимость должны происходить из одной таблицы покрытия, чтобы маршрут не
предпочитал поверхность, на которой затем движение неожиданно медленнее.

## Контракт маршрутизации

Новый публичный API должен развиваться от текущего `find_route(from, destination)`
к типизированному запросу:

```gdscript
class_name RouteRequest
extends RefCounted

var from: Vector3
var destination: Vector3
var traveler_profile: StringName = &"pedestrian"
var allow_destination_cell: bool = false
```

`traveler_profile` обязателен до появления велосипедов, тележек и машин. Он
задаёт доступность покрытия и его вес. Например, тропинка допустима для пешехода,
но не для тележки; грунтовая дорога допустима для тележки, но может иметь штраф
для машины в дождь. Нельзя кодировать эти ограничения только в визуальном типе
дороги или в логике актора.

`RouteResult` содержит `reachable`, `waypoints`, `arrival_position` и
`grid_revision`. Пустой список waypoints никогда не означает успешный маршрут.

## Владение данными

`NavGrid` — единственный источник проходимости и итогового веса, который читает
`GridRouteService`. `RoadNetworkService` владеет завершёнными дорожными
покрытиями и применяет их через API `NavGrid`:

```text
RoadNetworkService -> NavGrid.set_road_cell_weights(overrides) -> GridRouteService
```

`NavGrid` хранит независимые слои terrain overrides, pedestrian desire trails и
constructed-road overrides. Приоритет чтения: completed road -> profile trail
-> terrain -> grass. Поэтому демонтаж дороги вновь открывает ещё не заросшую
тропинку, а не уничтожает её данные. Bootstrap публикует только world facts;
`NavigationObstaclePublisher` превращает terrain и building footprints в
blocked cells.

Routing вынесен в отдельную feature `routing` (`game/features/routing/`), что
соответствует `docs/architecture.md`: выбор маршрута не зависит от `Citizen`, UI
или bootstrap-контроллера.

## Дороги

| Тип | Пешеход | Транспорт | Эра |
| --- | ---: | ---: | --- |
| Тропинка | 100% | запрещён | `Era.TENT` |
| Грунтовая | 100% | велосипеды, тележки | `Era.EARTH` |
| Глинобитная | 110% | грузовые велосипеды | `Era.CLAY` |
| Деревянная | 120% | мопеды | `Era.WOOD` |
| Каменная | 130% | ранний лёгкий транспорт | `Era.STONE` |
| Асфальтовая | 140% | лёгкий моторный транспорт | `Era.BRICK` |
| Асфальто-бетонная | 150% | тяжёлый транспорт | `Era.INDUSTRIAL` |

`Era.ECO` пока указана только как будущая эпоха после кирпичной. Эко-полосы и
эко-плитка не имеют отдельного `RoadType` и не участвуют в текущем алгоритме
дорог до отдельной проработки.

При ручной постройке клетка остаётся исходным покрытием до завершения работ.
После завершения `RoadNetworkService` меняет override веса, визуал и revision
сетки одной транзакцией. Улучшение дороги использует тот же процесс и не меняет
проходимость мгновенно.

## Desire lines и трафик

В начале палаточной эры поляна покрыта травой. Она проходима, но имеет вес `2.0`,
поэтому жители двигаются по ней вдвое медленнее базовой скорости. Тропинки не
строятся игроком: они возникают там, где жители действительно ходят. Это делает
первые маршруты медленными и постепенно формирует понятную, живую сеть между
костром, палатками, складом, рабочими местами и часто посещаемыми ресурсами.

Визуальный след тропинки непрерывен: он ставится вдоль фактического отрезка
движения жителя и не обязан проходить через центры клеток. Навигационное влияние
следа дискретно, потому что текущий маршрутизатор работает через `NavGrid`.
Поэтому сервис тропинок агрегирует пройденные отрезки в затронутые nav-клетки и
начисляет трафик только при входе жителя в новую nav-клетку, не на каждом кадре и
не на каждом физическом шаге. Это исключает зависимость от FPS и скорости
симуляции.

```text
record_walker_position(walker_id, position, road_walking)
  -> если горизонтальное смещение >= SAMPLE_DISTANCE:
    -> stamp_segment(from, to, stamp_strength)      # визуальный след
    -> для каждой новой nav-клетки на отрезке:
      -> traffic[cell] += traffic_strength * (2.0 if road_walking else 1.0)
      -> обновить навигационное состояние тропинки для этой клетки
```

`movement_mode` не является отдельным маршрутизатором и не меняет доступность
клеток. Он влияет только на то, насколько заметно житель протаптывает маршрут.
Обычное движение использует множитель `1.0`; режим «ходить как по дороге» —
повышенный множитель. Настройка относится к приказу поселения, а не к отдельному
жителю, поэтому все жители используют одинаковое правило, пока приказ активен.
Это постоянная глобальная политика костра, а не дневный приказ отдельным
жителям: она сохраняется между днями, пока игрок не выключит ее вручную.

Тепловая карта визуала хранится отдельно от nav-состояния и может быть плотной
текстурой или разреженным полем. Nav-состояние разреженное: `traffic[cell]`
хранится только для клеток, где был заметный поток. Раз в игровые сутки значения
умножаются на `1 - decay_rate`; значения ниже epsilon удаляются. Для
автопревращения в тропинку и деградации нужны разные пороги и задержка в
несколько дней, иначе покрытие будет мигать около границы:

- `path_create_threshold` переводит траву в молодую тропинку;
- `path_mature_threshold` переводит молодую тропинку в полноценную;
- `path_degrade_threshold < path_create_threshold` начинает деградацию;
- деградация завершается только после `N` подряд игровых дней ниже порога.

### Тропинки палаточной эры

В `Era.TENT` игрок не может размещать чертежи дорог, улучшать тропинки вручную
или расходовать на них ресурсы. Единственное дорожное покрытие эпохи — тропинка,
созданная трафиком жителей. Визуально она является органической desire-line
полосой по фактическому движению жителей, а не клеточной плиткой. Для навигации
она проецируется в nav-клетки, через которые проходит накопленный след. Это
покрытие доступно только пешеходному профилю и не считается дорогой для
транспорта.

Состояния клетки:

| Состояние | Условие | Вес | Визуал |
| --- | --- | ---: | --- |
| Трава | Трафик ниже порога создания | 2.0 | Обычная трава |
| Молодая тропинка | Достигнут `path_create_threshold` | 1.7 | Узкая полупрозрачная бежевая полоса поверх травы |
| Полноценная тропинка | Достигнут `path_mature_threshold` | 1.4 | Непрозрачная бежевая полоса по протоптанной линии |
| Зарастающая тропинка | После задержки ниже порога деградации | 1.7 -> 2.0 | Бежевый слой постепенно бледнеет и сужается |

Переход между состояниями плавный визуально, но навигационный вес меняется только
на указанных порогах. Визуальная ширина и непрозрачность зависят от нормированного
`path_strength` внутри клетки: след сначала проявляется бледной тонкой линией,
затем становится шире и заметнее. Максимальная ширина ограничена шириной
пешеходной тропы; она не превращается в клеточную дорогу и не расширяется от
интенсивного потока.

Пока трафик поддерживает `path_strength`, тропинка сохраняется. Если жители долго
не ходят по ней, суточное затухание снижает силу следа. После `N` подряд игровых
дней ниже `path_degrade_threshold` клетка становится зарастающей, а затем
возвращается к траве и удаляет override веса из `NavGrid`. Любой новый проход до
окончательного исчезновения повышает силу следа и отменяет деградацию.

### Приказ «Ходить как по дорогам»

У костра есть кнопка приказов, открывающая модальное окно приказов
поселения. В палаточной эре в нём доступен переключатель **«Ходить как по
дорогам»**. Он не рисует путь и не создаёт покрытие мгновенно: приказ велит
жителям при обычных переходах двигаться более собранно и сильнее протаптывать
выбранные маршруты.

- Включённый приказ устанавливает `movement_mode = road_walking` для новых и
  продолжающихся маршрутов жителей; начисление трафика использует повышенный
  `movement_mode_traffic_multiplier`.
- Приказ не отменяет критические потребности, прямое управление, запреты
  проходимости и другие правила `OrderBoard`; он является глобальной политикой
  движения, применяемой актуатором при исполнении маршрута.
- Переключение применяется сразу к последующим переходам клеток. Уже созданные
  тропинки не получают силу задним числом.
- Приказ сохраняется между днями и выключается только вручную через меню
  приказов костра.
- В модальном окне показываются активное состояние приказа и краткий эффект:
  жители быстрее формируют тропы, но не получают дополнительной скорости на
  траве до появления самой тропинки.

После появления тропинки маршрут уже сам предпочитает её: A* минимизирует
стоимость переходов, а вес тропинки ниже веса травы. Поэтому жители выбирают
немного более длинный, но более лёгкий маршрут по существующим тропам, продолжают
его протаптывать и стабилизируют полезные направления. Если обход по тропинке
дороже прямого пути по траве, выбирается прямой путь; система не должна
искусственно заставлять жителя делать неразумный крюк.

Автоулучшение грунтовой дороги создаёт обычный строительный заказ с ограничением
дневного бюджета, приоритетом ниже жизненно важных работ и явной проверкой эпохи,
исследований и ресурсов. Каменные и более качественные дороги сами не деградируют.

## Производительность и надёжность

- Минимальный вес кэшируется в `NavGrid` и пересчитывается только при изменении
  overrides.
- Для A* используется бинарная очередь; поиск не должен делать линейный обход
  очереди или всей сетки на каждом соседнем переходе.
- Очередь route requests должна иметь бюджет на кадр, когда население вырастет.
  Одинаковые запросы к той же цели можно кэшировать по revision.
- При изменении revision активные и отрицательные маршруты устаревают сразу.
- Сглаживание допускается только по проверенному `segment_cost`; полученные
  waypoints не являются достаточной проверкой безопасности сами по себе.

## Тесты

Минимальный набор автоматических проверок:

- достижимый и недостижимый маршрут;
- запрет диагонального срезания угла;
- выбор более длинного, но дешёвого коридора;
- сохранение корректных сегментов после сглаживания;
- изменение blocked cells увеличивает `topology_revision` и делает активный
  `RouteResult` небезопасным; изменение веса увеличивает только `revision` и
  применяется при следующем выборе маршрута;
- повторная установка идентичных данных не меняет revision;
- производительный маршрут на максимальном размере игрового поля;
- недостижимый маршрут не выполняется повторно до изменения revision или таймера.
- трафик начисляется ровно один раз при входе в клетку и не зависит от FPS;
- обычные проходы создают молодую и затем полноценную тропинку только после
  соответствующих порогов, а приказ `road_walking` ускоряет этот набор;
- тропинка имеет ширину одной клетки, снижает маршрутную стоимость и не доступна
  непешеходному профилю в `Era.TENT`;
- неиспользуемая тропинка проходит задержку деградации, удаляет override веса и
  возвращается к траве; новый проход до удаления отменяет деградацию.
