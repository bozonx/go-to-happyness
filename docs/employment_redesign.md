# Переработка системы найма (employment redesign)

Статус: **проект / план**. Дата: 2026-07-12.

Цель: убрать невидимого «фрилансера», сделать роли легибельными для игрока,
свести ~9 полей роли на жителе к 4, ввести дом стройматериалов как здание
профессии-собирателя, а ранний сбор оставить ручными приказами. Параллельно —
поправить баланс сбора на первые постройки и добавить герою роль «ускорителя».

---

## 1. Проблема (что не так сейчас)

- На жителе живёт 8–9 пересекающихся полей роли: `specialization`,
  `permanent_role`, `freelance_assignment`, `manual_role`, `active_role`,
  `training_role`, `temp_training_role`, `pending_employment_role`,
  `preferred_role()` + enum `EmploymentState {UNREGISTERED, FREELANCE, EMPLOYED,
  REGISTERING}`. Отсюда почти все прошлые баги («затупают», «не нанимаются»,
  RESTING удовлетворял цель Work).
- «Фрилансер» невидимо прыгает между ролями по скрытым очкам дефицита
  (`workforce_policy`). Игрок не понимает, почему работник сменил задачу.
- Экономического смысла у деления «фрилансер/постоянный» **нет** (у рабочих нет
  зарплат — деньги только на здания/инструменты/торговлю). В палаточной эре
  действует «общак», экономика найма не нужна вообще.
- Курьер — это костыль `freelance_assignment == "courier"` со спец-обработкой в
  `workforce_coordinator`, `trade_service`, `courier_dispatcher`.

Замечание по факту кода: **полевая регистрация уже убрана** и это доделано.
`_employment_center_position()` → `INF` без костра; на спавне (settlement_game.gd
~2022) до костра жители появляются сразу `FREELANCE`. Герой держит титул
`official`, но он инертен без гражданского центра. Осталась только концептуальная
путаница между `FREELANCE` и `UNREGISTERED`.

---

## 2. Целевая модель

Два ясных класса жителя вместо размытого «фрилансера»:

1. **Нанятый на рабочее место.** Есть профессия, привязан к зданию, автоматически
   делает работу этого здания (повар — кухня, лесоруб — лесопилка, курьер —
   логистический пост, собиратель — дом стройматериалов). «Автоматический», но
   **видно почему**: стоит у своего здания.
2. **Безработный.** Никуда не приписан. По умолчанию простаивает/отдыхает, пока
   **чиновник (игрок) не даст явный приказ** — например «собирать ветки».

**Авто-раздача = «чиновник планирует».** Скоринг из `workforce_policy` перестаёт
быть умолчанием и становится диегетическим: это твой чиновник распределяет людей.
Играешь героем-чиновником → раздаёшь сам. Не играешь → он раздаёт по скорингу.
Тем самым «почему Вася ушёл на ферму» имеет ответ, а не выглядит магией.

**Чиновник нужен с самого начала** — он и есть «планировщик», представитель
игрока. До костра титул инертен, но как только костёр/ратуша построены, чиновник
получает рабочее место и планирует эффективнее.

---

## 3. Упрощение статусов: 9 полей → 4

Статус занятости **выводится**, а не хранится.

| Было | Стало | Смысл |
|---|---|---|
| `specialization` + `preferred_role()` | **`profession`** | Карьера-идентичность (вкл. курьера, собирателя). Где растёт навык. |
| `permanent_role` + `employment_workplace` | **`workplace`** | Здание, где работает. Роль выводится из типа здания. `null` = безработный. |
| `freelance_assignment` + `manual_role` + пиннинг | **`order`** | Явный приказ чиновника (напр. «собирать ветки»). Опционален. |
| `active_role` | **`current_task`** | Что делает прямо сейчас (анимация/логика). Производное. |
| `training_role` + `temp_training_role` + `pending` | **`training`** | `{role, days}` или пусто. |

`EmploymentState` вычисляется:
- `workplace != null` → нанят;
- `order != null` → выполняет приказ;
- иначе → безработный.

`UNREGISTERED / REGISTERING` больше **не состояния рабочего**. Регистрация — это
просто действие чиновника «назначить»: короткая анимация планирования, не гейт.
Визуал «идёт к костру оформиться» можно оставить как флейвор поверх
`order`/`workplace`, но не как блокирующее предусловие.

---

## 4. Курьер — полноценная профессия

- Убрать спец-случай `freelance_assignment == "courier"` из
  `workforce_coordinator`, `trade_service`, `courier_dispatcher`.
- Курьер получает `profession = "courier"` и рабочее место (логистический пост /
  склад). Работает автоматически, как повар на кухне.
- Снаряжение курьера (`COURIER_EQUIPMENT`: рюкзаки/велосипед) остаётся — это
  прокачка профессии, а не признак «фрилансера».

---

## 5. Дом стройматериалов (двор собирателя)

Строится **в каждой эре**, даёт постоянную работу собирателям:

- Палатка: собирают **ветки + траву**.
- Земляная: **грунт (soil)** + ветки.
- Глиняная: **глина**.
- Каменная: **камень**.
- (Дерево: доски/брёвна уже покрыты лесопилкой — двор ориентируем на «сырьё
  земли».)

Собиратель во дворе — обычная профессия (`profession = "gatherer"`), роль
выводится из эры/типа двора. Двор задаёт, что именно собирать, и куда сдавать.

### Бутстрап: пока двора нет

Один глагол «собирать» в двух формах:

- **Ранняя (двора нет):** чиновник даёт безработным `order = "gather_branches"` /
  `"gather_grass"`. Здание не нужно — это первое действие игрока.
- **Поздняя (двор построен):** сбор становится постоянной работой во дворе;
  нанимаешь собирателей туда, приказы больше не нужны.

Это обучающая арка: сначала гоняешь людей вручную → строишь двор → автоматизируешь.

## 6. Баланс раннего сбора (чтобы не превратить в пытку)

### Замер сейчас
- День = 300 c реального; рабочий день 8 ч ≈ **100 c работы/день**.
- Новичок в палатке: КПД ≈ 0.85 → `carried_amount = 1` (порог 2 штук — `eff ≥
  1.05`). Цикл ≈ 12–15 c → **~6–8 единиц/работник/день**.
- Герой занят чиновником → собирают ~3 человека → **~20 единиц/день**.
- Первые постройки: костёр 6 веток; полноценный палаточный набор (костёр +
  палатки + кухня + forager + сборщик росы) ≈ **60+ веток, 40+ травы ≈ 3–5 дней**
  сбора по 1 штуке. Это и есть риск «пытки».

### Правки баланса
1. **Ранний сбор носит 2, а не 1.** Для сырья `branches`/`grass` в палаточной эре
   убрать зависимость `carried_amount` от `eff ≥ 1.05` (новички носят 2). Это
   вдвое сокращает самый скучный этап. Навык тогда влияет на *скорость цикла*, а
   не на «1 vs 2», и остаётся значимым в поздних эрах.
2. **Стартовый запас, чтобы показать петлю без гринда.** Дать `branches = 6` на
   старте → костёр строится сразу, игрок видит цикл, а не 1.5 дня копит на первое
   здание. (Либо снизить стоимость костра до 4.)
3. **Пересмотреть стоимость первого набора.** Ориентир: полноценный палаточный
   посёлок должен собираться за **~2 дня**, а не 3–5. Точки: костёр, 1 палатка,
   cook_campfire, forager_tent. Урезать на ~20–30% ветки/траву в этих 4 рецептах
   ИЛИ поднять выход сбора (см. п.1).
4. **Проверить дистанции спавна деревьев/травы** — ходьба доминирует в цикле
   (WALK_SPEED 2.2). Если источники далеко, никакой выход за цикл не спасёт.

Исходные данные рецептов — `building_catalog.gd`; выход сбора — `carried_amount` и
`WORK_DURATION` в `citizen_actor.gd`.

## 7. Герой как «ускоритель» (стимул играть руками)

Идея: NPC делают всё сами потихоньку; герой позволяет форсировать нужное
направление, если игрок готов поиграть.

- **Мульти-сбор.** Герой за один выход набирает **несколько единиц и разных
  материалов** (напр. ветки + трава в одну ходку) и приносит сам — в отличие от
  NPC (1–2 одного вида).
- **Больше ёмкость.** Герою — повышенная переноска (напр. 4–6 против 1–2 у NPC).
- **Усиление направления.** Играя героем, игрок точечно закрывает дефицит быстрее,
  чем автоскоринг чиновника. Это и есть смысл ручного управления: не обязателен,
  но ускоряет то, что иначе идёт медленно.
- Синергия с балансом: ручной герой — «клапан» против раннего гринда. Если п.6
  делает автосбор терпимым, герой делает его быстрым для тех, кто хочет.

## 8. Порядок рефактора (чтобы не сломать текущие фиксы)

Опираться на memory: [[work-rest-scheduling]], [[employment-officer-overhaul]],
[[tent-era-overhaul-plan]] — не сломать WAITING/RESTING-планирование и
storage-геймплей.

1. **Ввести `profession` / `workplace` / `order` / `current_task` / `training`**
   как новые поля, временно синхронизируя со старыми (adapter-слой), чтобы
   `workforce_policy` продолжал работать на словарях `_worker_data`.
2. **Курьер → профессия.** Заменить `freelance_assignment == "courier"` на
   `profession == "courier"` + рабочее место; вычистить спец-случаи.
3. **Дом стройматериалов.** Добавить здание в `building_catalog` + blueprint;
   регистрация как workplace типа «gatherer_yard»; роль сбора по эре.
4. **Ранний приказ на сбор.** UI/действие чиновника «назначить сбор» для
   безработного → `order`. Убрать авто-freelance-скоринг как умолчание (оставить
   как режим «чиновник планирует»).
5. **Схлопнуть `EmploymentState`.** Вывести статус из полей; удалить
   `UNREGISTERED/REGISTERING` как состояния (регистрация → действие-анимация).
6. **Баланс** (п.6) отдельным коммитом после того, как петля работает.
7. **Герой-ускоритель** (п.7) — последним, поверх стабильной петли.

Каждый шаг проверять: `godot --headless --check-only`, `tests/test_domain.gd`
(exit 0), headless-запуск сцены без SCRIPT ERROR, и поведенческий смоук (сбор →
первое здание → наём собирателя во двор).
