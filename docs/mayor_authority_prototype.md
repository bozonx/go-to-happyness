# Прототип: авторитет мэра/чиновника над трудом

Статус: **план**. Дата: 2026-07-12.

Цель — дёшево проверить, весёлый ли размен **«сам рулю трудом» ↔ «делегировал
NPC, ушёл героем в поле»**, не строя полную двухчиновничью систему и ИИ-застройки.
Стройка **остаётся полностью на игроке** (не гейтим). Опирается на уже
существующий officer-gate и единственную роль `official`.

## Суть (одно правило)

Ручные команды по труду доступны, **только когда officer-роль держит герой**.
Назначил officer'ом NPC → это «авто-режим»: NPC-чиновник ведёт план сам, ручные
команды заблокированы. Вернуть ручное управление — назначить officer'ом снова
героя.

Ключ: **officer в игре ровно один** (`_appoint_official` уже переназначает роль,
снимая её с предыдущего). Значит «кто держит роль» — это и есть переключатель
режима. Никакой второй роли для прототипа не заводим.

## Три состояния

| Кто officer | Ручные команды по труду | Авто-план (резерв→работа) | Стройка |
|---|---|---|---|
| **Герой** (по умолчанию) | ✅ доступны | идёт (или пауза, см. ниже) | ✅ |
| **NPC** (делегировано) | ❌ заблокированы, «авто» | идёт (ведёт NPC) | ✅ |
| **Никто** (NPC умер/снят, героя не назначили) | ❌ | ⛔ резерв простаивает | ✅ |

Авто-план резерва уже завязан на `_officer_available()` (роль существует), так что
**софтлока нет**: пока кто-то держит офис, работа идёт. Отдельно гейтим только
*ручные* команды игрока.

Опция (решить на плейтесте): когда officer = герой (ручной режим), **ставить
авто-план на паузу** для не-приказанных жителей, чтобы игрок реально командовал, а
не боролся с автоскорером. Проще для первого прототипа — НЕ ставить паузу
(авто-план и ручные приказы сосуществуют, как сейчас); включить паузу, если
почувствуется, что автомат «перебивает» игрока.

## Что гейтим, что нет

**Гейтим `_player_can_command_labor()` (= герой держит officer):**
- Приказы/пины/собирательство: `_set_manual_role(role)` — **кроме** `role ==
  "official"` (назначение officer'а всегда доступно, иначе не делегировать/не
  вернуть управление).
- Назначение специалистов на рабочие места: `_set_manual_specialist_employment`.
- Управление работниками у зданий и у костра: assign / accept / dismiss
  (`_assign_official_at_campfire` — это назначение officer'а, остаётся свободным;
  а `_toggle/_dismiss_selected_workplace_worker`, `_dismiss_campfire_worker`,
  `_toggle_campfire_acceptance` — гейтим).
- Обучение/смена профессии: точка старта обучения (`start_training`/attend school).
- Назначение раскопок: `_start_dig_assignment`.

**НЕ гейтим (всегда доступно):**
- **Строительство** (build-режим, кнопки, снос) — целиком на игроке.
- **Назначение officer'а** (`_appoint_official`, `_set_manual_role("official")`,
  `_assign_official_at_campfire`) — это governance, им переключаешь режим.
- Просмотр меню/выбор жителей (только чтение).

## Бутстрап и ранняя игра

- До первого костра/ратуши officer-роль у героя по умолчанию → ручные команды
  доступны, как сейчас. Ничего не ломается.
- Если officer'а нет вообще (редкий случай) и центра нет — команды доступны
  (founder-режим), чтобы не залочить старт. Практически: разрешать ручные
  команды, если `hero_citizen` держит роль **ИЛИ** officer'а нет и костра нет.

## Сигнализация (обязательно, иначе «почему кнопки серые?»)

- Если команда заблокирована (делегировано NPC): тост/лейбл
  «Труд делегирован: <имя NPC> управляет автоматически. Назначьте мэром героя,
  чтобы командовать вручную.»
- Кнопки труда в меню — **видимо disabled** с подсказкой, а не просто молчат.
- Если officer'а нет совсем: заметный индикатор «Посёлок без управления трудом:
  назначьте officer'а». (Стройка при этом работает — подсказать, что можно.)

## Точки в коде (settlement_game.gd)

1. Новый предикат:
   ```
   func _player_can_command_labor() -> bool:
       if is_instance_valid(hero_citizen) and hero_citizen.permanent_role == "official":
           return true
       # founder-режим до появления администрации
       return not _officer_exists() and _employment_center_position() == Vector3.INF
   ```
   (`_officer_exists()` = есть ли житель с `permanent_role == "official"`; можно
   переиспользовать логику `workforce._officer_available()`.)
2. В начале каждой гейтимой команды — ранний выход с тостом:
   ```
   if not _player_can_command_labor():
       _update_interface("Труд делегирован NPC-мэру. Назначьте героя, чтобы командовать.")
       return
   ```
3. В построении меню труда — ставить кнопкам `disabled = not _player_can_command_labor()`.
4. Стройку и `_appoint_official`/назначение officer'а НЕ трогаем.

## Как проверять (headless + плейтест)

- Headless: юнит-тест на `_player_can_command_labor()` в трёх состояниях (герой =
  officer → true; NPC = officer → false; нет officer + нет костра → true).
- Плейтест (главное — это про «весело ли»):
  1. Старт: команды доступны (герой — officer). Ок.
  2. Назначить NPC мэром → команды труда блокируются, идёт авто. Уйти героем
     собирать/строить. Приятно ли «отпустить руль»?
  3. Вернуть героя в officer → команды снова доступны. Понятен ли переход?
  4. Снять NPC, не назначив никого → резерв встал, явный индикатор, стройка
     работает. Понятно ли, что делать?

**Критерий успеха:** переключение режимов ощущается как осмысленный выбор
внимания, а не как «серые кнопки». Если да — только тогда думаем о v2.

## v2 (только если прототип зашёл) — НЕ делаем сейчас

- **Физический пост**: требовать, чтобы герой-officer был у ратуши/костра (уже
  есть `_registration_official()` с `OFFICER_POST_RADIUS`) — больше диегетики,
  больше трения. Включать тумблером.
- **Второй чиновник (мэр отдельно от службы занятости)** с чистым делением
  «труд vs стройка/стратегия» и **ценой в население** (полная автоматизация = 2
  админа = 2 непроизводящие головы).
- **Очередь строительства** для делегированного мэра (чтобы не строить ИИ-застройки
  с нуля — он исполняет заданные игроком приоритеты).
- **Пауза авто-плана** в ручном режиме, если автоскорер мешает микроменеджменту.
